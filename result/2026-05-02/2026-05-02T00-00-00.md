## 1. Project Overview

Go REST API for Sovra's subject data service. Manages Subjects, Properties, and Records used to derive those properties. Server-to-server only -- never called directly from the web app.

**Tech Stack:**
- Go 1.26.1, chi v5 (HTTP routing)
- PostgreSQL (production) / SQLite (development)
- sqlx + golang-migrate for DB access and migrations
- swaggo/swag for Swagger/OpenAPI docs
- Pulumi (Go) for AWS infrastructure
- Deployment target: AWS ECS Fargate (us-east-2)

## 3. Application Flow

```
main.go
  -> Initialize slog JSON logger
  -> Read env vars (PORT, DB_DRIVER, DATABASE_URL, AUTH_TOKENS, CORS_ALLOW_ORIGINS)
  -> Open database (SQLite or PostgreSQL via DB_DRIVER)
  -> Run migrations (golang-migrate)
  -> Create repositories (subjectRepo, recordsRepo, ratingsRepo, featuresRepo)
  -> Create services (SubjectService, RecordService)
  -> Create router (chi) with middleware (CORS, BearerAuth)
  -> Start HTTP server with graceful shutdown on SIGINT/SIGTERM
```

**Request lifecycle:**
```
Request -> chi.Router -> CORS middleware -> BearerAuth middleware -> Handler -> Service -> Repository -> Database
```

## 4. API Endpoints

### Authenticated (Bearer token via `/v1/*`)

| Method | Path | Handler | Description |
|--------|------|---------|-------------|
| GET | `/v1/subjects` | ListSubjects | List all subjects (ID + name) |
| POST | `/v1/subjects` | CreateSubject | Create new subject |
| GET | `/v1/subjects/{subjectID}` | GetSubject | Get subject with nested records/ratings/features |
| PUT | `/v1/subjects/{subjectID}` | UpdateSubject | Update subject name |
| DELETE | `/v1/subjects/{subjectID}` | DeleteSubject | Delete subject (cascades) |
| GET | `/v1/subjects/{subjectID}/records` | ListRecordsBySubject | List records for a subject |
| GET | `/v1/records` | ListRecords | List all records (filter by `?subject_id=`) |
| POST | `/v1/records` | CreateRecord | Create new record |
| GET | `/v1/records/{recordID}` | GetRecord | Get record by ID |
| PUT | `/v1/records/{recordID}` | UpdateRecord | Update record |
| DELETE | `/v1/records/{recordID}` | DeleteRecord | Delete record |

## 5. Authentication

- **Mechanism:** Bearer token via `Authorization` header
- **Source:** `AUTH_TOKENS` env var (comma-separated)
- **Behavior:** If `AUTH_TOKENS` is empty, auth is disabled (local dev convenience)
- **Implementation:** Map-based token lookup in `internal/middleware/auth.go`
- **Production:** Tokens stored in AWS Secrets Manager

## 6. Database Schema

```sql
subjects (id TEXT PK, subject_name TEXT UNIQUE, data TEXT [JSON], created_at, updated_at)
records (id TEXT PK, subject_id TEXT FK->subjects, data TEXT [JSON], created_at, updated_at)
subject_ratings (id TEXT PK, subject_id TEXT FK->subjects, data TEXT [JSON], created_at, updated_at)
subject_ce_features (id TEXT PK, subject_id TEXT, subject_ratings_id TEXT, data TEXT [JSON], created_at, updated_at)
```

- JSON blob storage pattern (data stored as TEXT, deserialized in Go)
- Cascading deletes from subjects to records/ratings
- Indexes on `subject_id` columns

## 7. Data Model (data-model module)

- **Subject:** id, subject_name, trace_data (records), ratings
- **Record (TraceData):** metadata (website, URL, capture_date, tags, identifiers), data_source (platform, content_type), platform_data (normalized, raw), images, videos
- **SubjectRatings:** rater_id, rating_timestamp, feature_ratings
- **SubjectCEFeature:** feature, category, rating (L/M/H), rating_as_number (1-5), rating_reasoning, confidence, evidence

## 11. Code Quality Assessment

### Strengths

- **Clean layered architecture** with dependency injection (handlers -> services -> persistence)
- **Proper error handling** with `%w` wrapping and early returns
- **Structured logging** via `slog` with JSON output
- **12-factor app** principles (env var config, stateless, port binding)
- **Multi-database support** (SQLite for dev, PostgreSQL for prod)
- **Comprehensive API docs** (Swagger UI + schema browser)
- **Graceful shutdown** with 10s timeout on SIGINT/SIGTERM
- **Parameterized SQL queries** (no SQL injection risk)
- **Infrastructure as Code** with Pulumi

### Issues

#### Critical
- **No test coverage** -- zero test files in the entire project

#### Security
- **No rate limiting** -- potential DoS vector
- **No request validation** -- missing field validation before DB operations
- **Bearer token lookup not constant-time** -- acceptable for small allowlists but not ideal

#### Performance
- **N+1 query problem** in `GetSubject` -- loads ratings, then loops to load features per rating
- **No pagination** on list endpoints -- returns all results

#### Data Integrity
- **No transaction support** -- multi-step operations (e.g., create subject + records) are not atomic
- **No optimistic locking** -- concurrent updates can silently overwrite each other

#### Error Handling
- **WriteJSON ignores encoding errors** (`handler.go`) -- incomplete responses if encoding fails
- **UpdateRecord masks error types** -- all errors reported as "not found"
- **No distinction between DB errors and not-found** in some handlers

#### Missing Features
- No pagination or filtering on list endpoints
- No soft deletes (permanent deletion only)
- No audit trail
- No request/response validation middleware

## 13. Recommendations

1. **Add tests** -- unit tests for handlers/services, integration tests for DB layer
2. **Add pagination** -- `?limit=&offset=` on all list endpoints
3. **Fix N+1 queries** -- batch load features with `WHERE subject_id IN (...)`
4. **Add request validation** -- validate required fields before DB calls
5. **Wrap multi-step ops in transactions** -- especially subject creation with nested data
6. **Add rate limiting** -- middleware for production deployment
7. **Improve error discrimination** -- distinguish not-found from DB errors in handlers
8. **Add soft deletes** -- `deleted_at` column for audit and recovery
