## 2. Entry Point (`cmd/run_service/main.go`)

**Initialization flow:**
1. JSON structured logging via `slog.NewJSONHandler` to stdout
2. Dual database support: SQLite (default, local dev) or PostgreSQL (production)
3. Embedded SQL migrations applied on startup via `golang-migrate/migrate/v4`
4. Dependency injection: Repositories -> Services -> Router -> HTTP Server
5. Graceful shutdown with 10-second timeout (SIGINT, SIGTERM)

**Environment variables:**

| Variable | Default | Purpose |
|----------|---------|---------|
| `PORT` | 8380 | HTTP listen port |
| `DB_DRIVER` | sqlite | `sqlite` or `postgres` |
| `DATABASE_URL` | subject-data.db | SQLite path or PostgreSQL DSN |
| `AUTH_TOKENS` | (empty) | Comma-separated bearer tokens |
| `CORS_ALLOW_ORIGINS` | (empty) | CORS origin (dev only) |

## 3. Internal Packages

### `internal/api/` - HTTP API Layer

**Files:** `server.go`, `router.go`, `handler.go`, `handler_subject.go`, `handler_record.go`, `models.go`, `types.go`

**Key types:**
- `Handler` - Holds logger, SubjectService, RecordService
- `ErrorResponse` / `ErrorDetail` - Standardized error JSON
- `SubjectSummary` - List response (ID + Name)
- `SubjectCreateRequest` / `SubjectUpdateRequest` - Mutation DTOs
- `StatusResponse` - Generic mutation result

### `internal/middleware/` - Authentication

**Bearer token authentication:**
- Extracts `Authorization: Bearer <token>` header (case-insensitive)
- O(1) token lookup via map
- Auth disabled if `AUTH_TOKENS` is empty (local dev convenience)
- Returns 401 JSON response on failure
- CORS middleware enabled conditionally via `CORS_ALLOW_ORIGINS`

### `internal/service/` - Business Logic

1. **SubjectService** - CRUD for subjects with nested records and ratings. Loads `JoinedSubjectData`, strips TraceData/Ratings before saving blob (stored separately in indexed repos). UUID v4 generation.
2. **RecordService** - CRUD for individual records. Filtering by `subject_id`. Auto-generates UUID.
3. **RatingStorageService** - Persistence for SubjectRatings and SubjectCEFeatures. Uses `INSERT ... ON CONFLICT` for upserts.

### `internal/persistence/` - Database Access

**Repository pattern with JSON blobs:**
- `Repository` - Generic CRUD: `id` (PK) + `data` (JSON TEXT) + timestamps
- `IndexedRepository` - Extends with secondary key columns (e.g., `subject_id`)
- `SubjectRepo` - Specialized for subjects table with `subject_name` column
- `.Rebind()` for cross-database SQL compatibility (? vs $1)

**Database drivers:**
- SQLite: `modernc.org/sqlite` (pure-Go, no CGO), WAL mode, 5s busy timeout
- PostgreSQL: `jackc/pgx/v5` with connection pooling

**Migrations:** Embedded SQL files, auto-fix dirty state, single schema version `001_initial_schema`

## 4. API Routes

### Health/Docs (No Auth)
```
GET  /healthz                            -> {status: "ok"}
GET  /readyz                             -> {status: "ok"}
GET  /docs/doc.json                      -> Swagger JSON
GET  /schema-browser/*                   -> Data model browser
```

### Subjects (Auth Required)
```
GET    /v1/subjects                      -> []SubjectSummary
POST   /v1/subjects                      -> StatusResponse{ID, Name}
GET    /v1/subjects/{subjectID}          -> JoinedSubjectData
PUT    /v1/subjects/{subjectID}          -> StatusResponse{ID, "updated"}
DELETE /v1/subjects/{subjectID}          -> StatusResponse{ID, "deleted"}
GET    /v1/subjects/{subjectID}/records  -> []Record
```

### Records (Auth Required)
```
GET    /v1/records                       -> []Record (filter: ?subject_id=X)
POST   /v1/records                       -> StatusResponse{ID}
GET    /v1/records/{recordID}            -> Record
PUT    /v1/records/{recordID}            -> StatusResponse{ID, "updated"}
DELETE /v1/records/{recordID}            -> StatusResponse{ID, "deleted"}
```

**HTTP codes:** 200 (GET/PUT), 201 (POST), 400, 401, 404, 500

## 5. Data Model (`data-model/`)

**Core types (~800 lines):**
- **Subject** - Person being analyzed (name)
- **Record** - External source data (platform, content, metadata, media)
- **SubjectRatings** - Assessment by a single rater
- **SubjectCEFeature** - Individual trait rating with evidence
- **JoinedSubjectData** - Subject + all records + all ratings

**Enumerations:**
- Platform: bluesky, crunchbase, facebook, instagram, linkedin, reddit, twitter, tiktok, transunion, etc.
- ContentType: post, profile, comment, reaction, document, court records, etc.
- Category: Normal Personality, Personality Under Stress, Character, Psychopathy, Values, Motivations, Aptitude, Risk Taxonomy, Personality Dx
- RatingStr: Low, Low/Moderate, Moderate, Moderate/High, High
- ConfidenceStr: Low, Medium, High

**Custom marshalers:**
- `DateTime` - Accepts ISO 8601 and human formats (e.g., "27 January 2026"), outputs ISO 8601 with ms precision

```
                    +------------------+

          |  |  RDS PostgreSQL               |   |
          |  |  db.t4g.micro, single-AZ      |   |
```

**Components:**
- **Networking** - VPC, 2 public subnets, IGW, security groups
- **RDS** - PostgreSQL (db.t4g.micro), custom subnet group, SG restricted to ECS
- **ECR** - Existing repository lookup
- **ECS Fargate** - Cluster, task def (2 containers), CloudWatch logs (30d retention)
- **API Gateway** - HTTP API, VPC Link, IAM auth, cross-account invoke roles
- **S3 + IAM** - KMS-encrypted bucket, ECS task role, IAM user with access key

## 10. Code Quality

### Strengths
1. Clean layered architecture (API -> Service -> Repository -> DB)
2. Dependency injection throughout (no globals)
3. Structured JSON logging via `slog`
4. Error wrapping with `%w`
5. Cross-database support (SQLite + PostgreSQL)
6. Graceful shutdown with signal handling
7. golangci-lint with 10+ linters enabled
8. Swagger docs auto-generated from comments
9. Pulumi IaC for repeatable deployments
10. Multi-stage Docker build for minimal images

### Weaknesses
1. **No test coverage** - Complete absence of tests
2. **Limited input validation** - Only SubjectName checked
3. **No request size limits** - Large payloads not guarded
4. **No rate limiting** - API Gateway not throttled
5. **No request/response logging middleware** - No metrics
6. **Hardcoded AZs** in Pulumi (us-east-2a, us-east-2b)
7. **JSON blob storage** - Loses RDBMS query flexibility

## 11. Recommendations

### High Priority
1. **Add comprehensive test suite** - Unit tests for handlers, services, middleware; integration tests with SQLite
2. **Input validation** - String lengths, enum values, record structure validation
3. **Request middleware** - Content-Type enforcement, request size limits, request ID tracing

### Medium Priority
4. **Pagination** - ListSubjects currently loads all records
5. **Rate limiting** - API Gateway throttling + per-token middleware
6. **Observability** - Request logging middleware, Prometheus metrics, distributed tracing
7. **Security** - API key rotation, request signing verification

### Low Priority
8. **Migration testing** - Validate up/down migrations
9. **Local dev** - docker-compose Postgres service, seed data scripts
10. **Pulumi CI** - GitHub Actions for preview/update, stack protection
